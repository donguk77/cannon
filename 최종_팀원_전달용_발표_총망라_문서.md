# 🚀 [팀원 전달용 완전판] 문제정의서 발표 및 아키텍처 총망라 기획 (Why-Monster 방어 메뉴얼)

💡 **문서의 목적**: 본 문서는 발표 시 교수님이나 심사위원이 **"근데 왜 하필 그 모델 썼어?", "왜 그 지표가 목표야?", "데이터도 없으면서 딥러닝이 돼?"**라고 끝없이 꼬리를 무는 **'왜몬(Why-Monster)'의 모든 공격을 논리적/수학적으로 완전하게 박살내기 위해**, 어떠한 생략도 없이 아주 치밀하고 과하다 싶을 정도로 모든 '선택의 이유(WHY)'를 구체적으로 풀어쓴 통합 해설서입니다. 프로젝트를 1도 모르는 사람이 읽어도 고개를 끄덕이게 만드는 기획안의 끝판왕입니다.

---

## 1. 🚨 기획의 출발점: "진짜 문제"의 재정의와 제약조건의 이해
> **왜 우리는 단순한 '성능 향상'이 아니라 '환경적 제약 극복'을 문제로 정의했는가?**

1. **데이터 희소성 (100장 미만)**
   - **WHY (왜 문제인가?)**: 딥러닝(CNN, ResNet 등)이 대상의 다양한 조명, 각도, 흔들림을 스스로 깨우치려면 최소 클래스당 수천~수만 장의 데이터가 필수적입니다. 하지만 산업 현장(공장)은 보안 지침이나 라인 교체 주기가 짧아 겨우 100장 남짓의 사진만 주어집니다.
   - **결론**: 애초에 무거운 딥러닝에게 1~4번의 헷갈리는 화면 종류를 다 외우라고 강요하는 것 자체가 **'과학적으로 불가능한 미션(Overfitting 확정)'**임을 인정하는 것에서 출발해야 합니다.
2. **유지보수 지옥 (소프트웨어 패치)**
   - **WHY (왜 문제인가?)**: 만약 수만 장을 모아서 인공지능을 깎아 놨다고 가정합시다. 그런데 다음 달에 캐논이 펌웨어 업데이트를 해서 UI 버튼 색이 파란색에서 빨간색으로 살짝 바뀌면? AI는 바보가 되고, 처음부터 수만 장을 다시 찍고 라벨링해야 합니다. 
   - **결론**: **가변적인 내부 UI 화면을 통째로 딥러닝에 학습시키는 짓은 기업 돈을 까먹는 끔찍한 행위**입니다.
3. **사용자의 비전문성 (Non-Expert)**
   - **WHY (왜 로봇이 아니라 작업자에 맞추는가?)**: 요구사항에 '로봇이 동작을 수행한다'고 되어 있지만, 로봇은 AI가 넘겨주는 True/False 전기 신호만 받아먹는 '수동적 기계'일 뿐입니다. 에러가 났을 때 프로그램을 리셋하고, 타겟 화면이 바뀌었을 때 학습 버튼을 누르는 **시스템의 핵심 관리자는 '컴퓨터를 모르는 공장 아저씨'**입니다. 터미널 창(CMD)에 파이썬을 치라고 하는 순간 이 시스템의 상용화는 실패합니다.

## 2. 🧠 핵심 해결책: 투트랙 듀얼 하이브리드 아키텍처 (YOLO + 제로샷 ORB/OCR)
> **수많은 인공지능 중에 왜 하필 '이 조합'을 융합시켜야만 했는가?**

### ① 1차 방어막: Custom YOLOv8n (껍데기 탐지 전담)
*   **왜 일반 CNN(분류 모델)이 아니라 YOLO(탐지 알고리즘)인가?** 
    - CNN은 사진을 통째로 보고 "어? 구석에 커피 컵이 있네? 이거 모니터 아니야"라고 오해(OOD 문제)합니다. 반면 YOLO는 현장의 노이즈(배경)가 어떻든 **"나는 파란색 박스로 캐논 모니터의 기계 테두리(베젤)만 물고 늘어진다"**는 록온(Lock-on)이 가능합니다.
*   **왜 데이터가 없는데 YOLO를 쓰는가?**
    - 모니터 기계의 '단단한 플라스틱 테두리'는 내부 UI가 바뀌어도 영원히 성질이 변하지 않습니다. 따라서 테두리만 외우는 데는 단 100장의 사진으로도 완벽한 과적합(최적화)이 가능하기 때문입니다.

### ② 2차 방어막: 제로샷(Zero-shot) ORB 특징점 혹은 파이썬 OCR 엔진
*   **왜 화면 내용물 판독을 딥러닝에 맡기지 않았는가?**
    - 딥러닝은 학습 비용이 무한대로 듭니다. 반면 ORB(수학 특징점)나 OCR(텍스트 추출)은 **'규칙(Rule) 기반 시스템'**입니다. 
    - **궁극의 유지보수 방어막**: UI 색상이나 디자인이 아무리 바뀌어도, 조작자가 폴더에 **'새로운 타겟 사진 원본 딱 1장(제로샷)'**만 끌어다 넣거나, "Next"라는 글자만 긁어오라고 설정하면 즉각 공장에 패치가 끝납니다. 유지보수 및 재학습 비용이 '0원'이 되기 때문에 딥러닝의 한계를 벗어나는 유일한 구원줄입니다.

## 3. 🎯 어떻게 이 모델이 훌륭함을 입증할 것인가? (정량적 목표 지표와 근거)
> **왜 평가지표를 분리했으며, 왜 하필 '이 수치'를 목표로 잡았는가?** (왜몬 방어 핵심)

### ① 하드웨어 탐지(YOLO) 지표: `mAP50 90% 이상`
*   **왜 mAP50인가?** AI가 친 네모 박스가 실제 정답과 50% 이상 겹치면(IoU>0.5) 정답으로 인정하는 자율주행계의 글로벌 스탠다드입니다. 우리는 모니터 화면 크기를 정확히 1픽셀 오차 없이 자르는 게 목표가 아닙니다. 화면의 대략적인 중심부로 렌즈를 끌고 들어가는(크롭) 역할이므로 mAP50이면 모니터 안쪽을 추출하는 데 100% 충분합니다.
*   **왜 99%가 아니라 90%인가?** 사람이 흔들리는 핸디 카메라로 촬영하기 때문입니다. 99%를 요구하면 AI는 인간의 미세한 손떨림을 '오류'로 판단해 로봇을 정지시켜 버립니다. 90%라는 수치는 **'가짜 엉뚱한 배경(예: 창문)은 완벽히 걸러내되, 약간의 카메라 흔들림은 너그럽게 포용하는 가장 현실적인 범용성 수치'**입니다.

### ② 소프트웨어 검증(ORB/OCR) 지표: `Recall(재현율) 75% 마지노선` 및 `초당 15FPS 확보`
*   **왜 Precision(정밀도)가 아니고 Recall(재현율)인가?** 
    - 공장에서는 "아침 햇살이 강해서 빛 반사(글레어)가 생겼다고 검사 시스템이 멈춰버리는 멍청함(미검출)"이 가장 혐오받습니다. 불량품을 흘려보내는 오탐도 문제지만, 정답이 눈앞에 있는데 렌즈 반사 때문에 못 찾는 치명적 문제를 방어하기 위해 "진짜 1번 화면이라면, 웬만큼 일그러져도 75% 확률로는 무조건 '가서 일해라'라고 알아먹는다"는 강건함(Robustness)의 보증 수표로 Recall 75%를 택했습니다.
*   **왜 최소 15 FPS 방어, 최종 목표 30 FPS 이상인가?** 
    - 초당 15프레임은 지연 속도로 치면 약 0.066초(66ms)입니다. 인간의 평균 시각 반응 속도가 0.25초(250ms) 내외임을 감안할 때, 15 FPS 이상이 방어되면 **인간 조작자의 눈에는 렉이 안 걸리는 부드러운 영상으로 착각되며 로봇을 전혀 딜레이 없이 통제 가능한 실무 마지노선**입니다. 단, 로봇의 정밀 제어를 위해 **최종 서비스 목표는 프레임 스킵 최적화를 적용하여 30 FPS 이상을 확보**하는 것으로 설정했습니다.

## 4. 🌔 투트랙(Dual-Track) 전략: 실시간 서비스와 오프라인 자가 진화 생태계
> **"속도(실시간)와 정확도(오프라인)를 하나의 모델로 잡는 건 물리적으로 불가능하다. 그래서 두 개로 나눈다."**

### ① [Track A] 실시간 서비스 파이프라인 (속도 우선: 65ms 이내)
현장에서 로봇을 쉬지 않고 돌리는 **'반사신경'** 역할. 가볍고 빠른 모델만 배치합니다.
- **YOLO → ORB/OCR**: 이미 위 2~3번 섹션에서 설명한, 65ms 이내의 초경량 듀얼 엔진.
- **원칙**: 이 파이프라인에는 **절대로 무거운 딥러닝 모델(비전 LLM, Siamese 등)을 올리지 않습니다.** 속도가 죽으면 로봇 제어가 죽기 때문입니다.

### ② [Track B] 오프라인 정밀 분석 파이프라인 (정확도 우선: 속도 무관)
실시간 서비스가 **"이거 1번인지 2번인지 애매한데?"**라고 판단을 보류한 사진들을 모아서, 퇴근 후 야간에 초정밀 분석하는 **'대뇌'** 역할. 속도 제한이 없으므로 무거운 모델을 자유롭게 투입합니다.

**[오프라인 폭포수(Waterfall) 판독 파이프라인]**

```
입력: 실시간 서비스가 "애매함"으로 분류한 사진 묶음
    │
    ▼
[1단계] YOLO → 모니터 베젤 크롭 (배경 제거)
    │
    ▼
[2단계] Siamese Network (Few-Shot DNA 감별)
    │ "타겟 원본 1~5장과 거리를 계산해서 1번/2번/3번/4번 판별"
    │
    ├── 확신도 90% 이상 → ✅ 정답 확정 → 자동 라벨링 완료
    │
    ├── 확신도 50~90% → ⚠️ 추가 검증 필요
    │   │
    │   ▼
    │   [3단계] LLM/VLM (의미론적 최종 판결)
    │   │ "이 화면에 'Next' 글자가 보이는가? 좌측 하단에 버튼이 있는가?"
    │   │ → 인간처럼 텍스트와 레이아웃을 '추론'하여 최종 결정
    │   │
    │   └── ✅ 정답 확정 → 자동 라벨링 완료
    │
    └── 확신도 50% 미만 → ❌ 판별 불가 → 사람에게 넘김 (Active Learning 큐)
```

*   **왜 샴(Siamese) 네트워크나 비전 LLM을 실시간(Track A)에 올리지 않았는가?**
    - 3주라는 극악의 개발 소요 시간적 한계와, 공장 엣지(Edge) 로컬 PC에는 고가의 그래픽 카드(VRAM 24GB 이상)가 아예 없다는 태생적 한계 때문입니다. 아무리 최적화해도 실시간 30프레임 속에서 저런 무거운 폭포수 파이프라인 연산을 즉각 돌리면 컴퓨터가 뻗거나 로봇이 즉사(?)합니다. 하지만 **오프라인 자가학습 큐(Track B)에서는 1초라는 시간 제한도 없으므로** 장당 10초가 걸리든 여유롭고 완벽에 가까운 초정밀 분류(Active Learning)가 가능합니다.

*   **그런데 샴 네트워크는 왜 가능한가? (딥러닝인데 데이터 많이 필요하지 않은가?)**
    - **핵심 무기 (Few-Shot Learning)**: 샴 네트워크는 일반 딥러닝처럼 "수만 장의 사진을 보고 외워라" 하는 모델이 아닙니다. 이 모델의 본질은 **"서로 다른 두 사진 사이의 DNA가 일치하는가(거리 유사도)?"**를 계산하는 수학(Metric Learning)입니다. 얼굴 인식(Face ID)처럼, 깨끗한 타겟 화면 **단 1~5장(Anchor)**만 뇌에 입력해 두면, 조명이나 노이즈로 개박살이 난 공장 사진을 보더라도 "원본이랑 DNA 거리가 가깝네! 너 1번!" 하고 감별해 냅니다.

*   **왜 이상탐지 전용 모델(PatchCore 등)을 기각하고 샴(Siamese) 네트워크를 선택했는가? (Why-Monster 방어용)**
    - PatchCore는 정상 이미지를 다수 학습하여 품질 불량을 찾는 데 특화되어 있지만, 무거운 특징 지도(Feature Bank)를 구축해야 하므로 오프라인일지라도 메모리 및 인퍼런스 리소스를 낭비합니다. 또한 우리의 핵심 문제는 단순한 "결함(Anomaly) 찾기"가 아니라, 이 공장 화면이 "1번 타겟 화면 원본과 얼마나 결이 같은가"를 **정확히 판별(Classification)**하는 것입니다.
    - 반면, **샴 네트워크(Siamese Network)**는 매우 가벼운 뼈대(ResNet18 등)만으로도 두 사진 간의 '거리(DNA 유사도)'를 빠르고 정확하게 계산해냅니다. 컴퓨팅 리소스 보호와 최종 화면 분류 정확도(Accuracy)라는 두 마리 토끼를 모두 잡기에 **샴 네트워크가 압도적인 이득(Trade-off 우위)**이라고 판단하여 PatchCore의 도입을 최종적으로 기각했습니다.

### ③ 유지보수 0원의 자가 진화 생태계 (왜 사람 손이 안 가는가?)

> **"한번 세팅해 놓으면 AI가 알아서 똑똑해진다"는 꿈같은 이야기의 구체적 설계도**

**[자동 학습(Self-Improvement) 루프 설계]**

| 단계 | 누가 하는가 | 무엇을 하는가 | 사람 개입 필요? |
| :--- | :--- | :--- | :--- |
| 1. 데이터 수집 | 실시간 서비스 (Track A) | "애매함" 판정 사진을 `/pending/` 폴더에 자동 저장 | ❌ 자동 |
| 2. 정밀 분류 | 오프라인 샴 네트워크 (Track B) | 확신도 90% 이상인 사진에 자동 라벨(1번/2번) 부착 | ❌ 자동 |
| 3. 최종 판결 | 오프라인 LLM/VLM (Track B) | 샴이 헷갈리는 사진을 의미론적으로 추론하여 라벨 확정 | ❌ 자동 |
| 4. 파라미터 튜닝 | Python 스크립트 (자동 배치) | 새로 분류된 데이터를 기반으로 ORB의 임계값(CLAHE 강도 2.0, 매칭 커트라인 25점 등)을 통계적으로 재조율 | ❌ 자동 |
| 5. 모델 갱신 | 작업자 (App Studio GUI) | "로봇 똑똑해지기(Train)" 버튼 클릭 1번 → 새 `.pt` 파일 적용 | ⚠️ 클릭 1번 |
| 6. 예외 처리 | 작업자 (App Studio GUI) | 샴+LLM도 판별 못한 사진만 직접 확인 (전체의 약 5% 미만 예상) | ⚠️ 극소량 |

**왜 유지보수가 거의 0원인가?**
*   기존 방식: 새로운 화면이 추가될 때마다 **사람이 수백~수천 장을 일일이 라벨링** → 모델 재학습 → 테스트 → 배포 (수일~수주 소요)
*   우리의 방식: 실시간 서비스가 자동으로 에러 데이터를 수집 → 샴/LLM이 야간에 자동 분류 → 통계 스크립트가 파라미터 자동 조율 → **작업자는 아침에 출근해서 "Train" 버튼 클릭 1번** (수 분 소요)
*   **결론**: 라벨링 인건비 0원, 전문 엔지니어 불필요, 비전문가 작업자의 클릭 1번으로 **AI가 매일 아침 스스로 진화하는 0원짜리 MLOps 생태계**가 완성됩니다.

## 5. 🛠️ 극한의 자원 낭비 방어: "결정적 꼼수 3대장 (최적화 방어 논리)"
> **현장 구형 컴퓨터와 무선 스트리밍의 성능 한계를 초과 달성하게 만든 마법들**

1. **프레임 스킵 (Frame-Skipping)**
   - **왜 했는가?**: 파이썬은 GIL이라는 병폐 탓에 멀티코어를 제대로 못 써서 연산이 버벅거립니다. 1초에 들어오는 30장의 사진에 무거운 YOLO 연산을 30번 넣으면 반드시 PC가 먹통이 됩니다.
   - **원리**: 1초에 단 10장만 풀파워 연산(Process 1)을 하고, 나머지 20장은 아예 뇌를 끄고 0ms로 도려냅니다(Skip 2). 이렇게 지연시간을 1/3로 깎아서 **어떠한 저사양 로컬 PC를 쓰더라도 FPS 하락이 절대 없도록 물리적 연산량을 증발**시켰습니다.
2. **좀비 스태빌라이저 (소프트웨어 잔상 유지)**
   - **왜 했는가?**: 프레임 스킵으로 연산을 쉬게 되면, 로봇에게 신호가 끊기거나 화면 박스가 미친 듯이 깜빡여 사용자를 혼란에 빠뜨립니다.
   - **원리**: 연산을 쉰 20프레임(0.5초) 동안만, 이전에 합격했던 초록색 정답 박스를 유령(고스트)처럼 화면에 스티커로 강제로 붙여 놓습니다. 로봇 제어와 시각적 부드러움을 공짜 연산으로 무사 통과시킵니다.
3. **사용자 유연성을 위한 '유한 상태 머신(FSM) 역순 롤백' 허용 (연산 비용 트레이드오프)**
   - **왜 했는가?**: 기업 요구사항이 '순차 검사(1→4)'라고 해서, 1번을 찾은 다음 눈을 가리고 오직 2번만 찾게 강제 코딩하면 비전문 작업자가 실수로 장비를 재부팅했을 때 로봇은 "난 2번 찾아야 하니까 1번 안 보여" 하고 공장을 세워버립니다. 인간의 실수를 포용하고 로봇을 되살리는 엣지 예외 통제망이 필수적입니다.
   - **연산량 증가에 대한 반박 (Why-Monster 방어)**: 심사위원이 *"과거 타겟(1번~3번)을 계속 같이 검색하면 연산량이 늘어서 렌즈가 버벅이지 않나?"* 라고 공격할 수 있습니다. 
   - **증명 논리**: 단일 1개의 목표만 찾을 때 매칭 연산은 약 `4.78 ms`가 걸립니다. 과거 화면까지 총 4개를 전부 뒤지면 약 `16.00 ms`가 소요됩니다. 즉 **불과 `11.22 ms`의 추가 연산이라는 아주 싼 값만 지불하면, 비전문가 작업자의 실수로 인한 시스템 블랙아웃(교착 상태)을 영구적으로 방어하는 무적의 안전망을 획득**하게 됩니다. 이 11ms의 지불은 평균 30FPS 방어 통제망 내부에서라면 무조건 투자해야 하는 가성비 최고의 '기획적 거래(Trade-off)'입니다.

## 6. 🖥️ 비전문가를 허용하는 유일한 해답: 'No-Code GUI 생태계 (App Studio)'
> **왜 로컬 PC에서 굳이 데스크톱 프로그램을 만들었는가? (조작자 배려)**

*   **문제점**: "동영상 넣어주세요", "파이썬 실행해 주세요"라고 공장 작업자에게 터미널 명령어를 치게 하는 순간 에러가 속출합니다. 유지보수의 주체는 비전문가입니다.
*   **해결책 (App Studio UI 아키텍처)**: 저희는 모든 AI 로직을 숨긴 직관적인 GUI 데스크톱 프로그램을 자체 개발했습니다.
    1. **[실시간 관제 탭 (모니터링)]**: 검은색 코드 창이 아니라, 초록색(정상)과 빨간색(미검출) 박스가 직관적으로 깜빡입니다. 우측 하단에는 **'현재 인식 FPS 지표(예: 30 FPS)'**와 **'ORB 일치율 퍼즐 개수'**가 실시간 그래프로 떠서 상태를 투명하게 보여줍니다.
    2. **[자가 MLOps 학습 탭 (비전문가 라벨링)]**: 새로운 화면이 나와 에러가 날 경우, 작업자는 "학습" 탭에 들어가 에러 동영상을 마우스로 드래그 앤 드롭합니다. 뒤에서 `video_to_frames.py`가 화면을 즉각 잘라주고, 마우스로 타겟 칸만 쓱(Drag) 그린 뒤 **"로봇 똑똑해지기(Train)"** 버튼만 누르면 백그라운드 모델이 갱신됩니다. 철저한 노-코드(No-code)의 실현입니다.
    3. **[E-Stop (긴급 정지 통신망)]**: AI의 딜레이가 500ms(0.5초) 이상 치솟는 등 화면 인식이 멈췄다고 판단되면, 즉시 UI에 붉은색 경고가 뜨고 로봇에게 **긴급 정지(E-Stop) 프로토콜**을 발송해 하드웨어 파손을 막아내는 안전망 메커니즘을 싣습니다.

### ✅ [실증 1] 환경이 바뀌어도 성능이 흔들리지 않는다 (C++ 코어 안정성)
> 두 가지 서로 다른 실행 환경에서 **완전히 동일한 코드**를 돌렸을 때의 실측값 비교

**🟡 가혹 조건 (VS Code Debugger 모드)이란?**
VS Code에서 `F5(Run and Debug)` 버튼을 눌러 실행하는 모드입니다. 이때 파이썬 내부에 **`debugpy`라는 추적 프로그램**이 함께 기동되어, 코드 한 줄 한 줄을 실시간으로 감시(Trace)합니다. 변수가 지금 몇인지, 에러가 어디서 나는지를 매 순간 기록하기 때문에, **순수 연산 외에도 파이썬 인터프리터 레벨에서 추가적인 오버헤드(감시 비용)**가 발생합니다. 개발 중에는 편리하지만, 성능 측정 시에는 실제보다 느린 결과가 나오는 '가혹한(부하가 높은)' 조건입니다.

**🟢 표준 환경 (Terminal 직접 실행)이란?**
터미널(CMD/PowerShell)에서 `py script.py`로 직접 실행하는 모드입니다. 디버거의 감시 프로그램 없이 **파이썬 인터프리터가 순수하게 코드만 실행**하기 때문에, 실제 공장 현장에서 프로그램을 `.exe`로 패키징하여 배포했을 때의 성능과 가장 유사한 조건입니다.

| 프로세스 스텝 | 기술 | 🟡 가혹 조건 (Debugger) | 🟢 표준 환경 (Terminal) | 차이 |
| :--- | :--- | :--- | :--- | :--- |
| 0. 전처리 | CLAHE + Sharpen | `1.30 ms` | `1.26 ms` | -3% |
| 1. YOLO 탐지 | YOLOv8n (CPU) | `32.38 ms` | `27.57 ms` | **-15%** |
| 2. ORB 특징점 | ORB_create(700) | `20.56 ms` | `18.15 ms` | -12% |
| 3. FSM 롤백 (4개) | KNN Matcher | `16.00 ms` | `16.07 ms` | ≈0% |
| 4. 렌더링 | GUI 송출 | `~2.00 ms` | `~2.00 ms` | 0% |
| **총계** | | **`72.24 ms` (13.8 FPS)** | **`65.06 ms` (15.4 FPS)** | **-10%** |

💡 **핵심 발견 (Why-Monster 방어용 심층 해설)**

두 환경 간 총 성능 차이는 **불과 약 10%**에 불과합니다. 디버거가 코드를 감시하는데도 왜 큰 차이가 안 나는 걸까요?

*   **이유**: 저희 파이프라인의 무거운 연산(YOLO 추론, ORB 특징점 추출, KNN 매칭)은 **파이썬으로 짠 코드가 아닙니다.** 내부적으로는 C/C++로 컴파일된 고성능 라이브러리(OpenCV, LibTorch)의 네이티브 바이너리가 실행됩니다. 디버거가 감시할 수 있는 것은 파이썬 레벨(함수 호출, 변수 할당)뿐이며, **C++ 코어 안에서 벌어지는 행렬 곱셈/특징점 계산은 디버거의 감시 범위 밖**이라 속도가 전혀 영향을 받지 않습니다.
*   **YOLO에서 15% 차이가 나는 이유**: YOLO 모델은 추론(C++ 코어) 전후에 파이썬 레벨에서 이미지 전처리(리사이즈, 정규화)와 후처리(NMS, 박스 변환)를 수행하는데, 이 **파이썬 래핑(Wrapper) 코드만** 디버거의 영향을 받아 약간 느려집니다.
*   **KNN 매칭이 0% 차이인 이유**: OpenCV의 `BFMatcher.knnMatch()`는 호출 시 **즉시 100% C++ 코어로 진입**하여 매칭 연산을 수행하고 결과만 파이썬에 반환하므로, 디버거 오버헤드가 사실상 0입니다.

**결론**: 디버거든 터미널이든 배포판이든, **어떤 실행 환경에 던져놓아도 항상 65~72ms 범위 안에서 일관된 성능을 보장**합니다. 이것은 산업 현장에서 가장 중요한 **'성능 예측 가능성(Predictability)'**을 달성했음을 의미하며, 공장 담당자가 "이 프로그램 깔았더니 갑자기 로봇이 느려졌다"라고 불평할 가능성을 원천 차단합니다.

├── gui/                               ← [프론트엔드] PyQt5 데스크톱 UI
│   ├── main_window.py                 ← 메인 윈도우 (탭 전환 컨트롤러)

    ├── fps_benchmark.py               ← 성능 계측 벤치마크 (이미 작성 완료)

### ② 백엔드 아키텍처 (Python 엔진 모듈 분리)

> **왜 하나의 거대한 `main.py`가 아니라 모듈별로 분리하는가?**
> 팀원 3명이 동시에 코드를 짜도 충돌이 안 나게 하기 위함입니다. 각자 담당 모듈만 수정하면 됩니다.

| 모듈 파일 | 담당 역할 | 입력 | 출력 |
| :--- | :--- | :--- | :--- |
| `preprocessor.py` | 영상 수신 + CLAHE + 샤프닝 | 웹캠 Raw Frame | 전처리된 Frame |
| `detector.py` | YOLO 베젤 탐지 + ROI 크롭 | 전처리된 Frame | 크롭된 모니터 영역 |
| `matcher.py` | ORB 특징점 + KNN 매칭 + FSM | 크롭 영역 + 타겟 이미지 | 매칭 결과 (화면 번호, 확신도) |
| `ocr_fallback.py` | ORB 실패 시 텍스트 기반 판별 | 크롭 영역 | 텍스트 추출 결과 |
| `frame_skipper.py` | 프레임 스킵 + 좀비 잔상 유지 | 매칭 결과 | 최종 로봇 제어 신호 |

**데이터 흐름 (1프레임 처리 순서)**:
```
웹캠 → preprocessor → detector → matcher → frame_skipper → GUI 표시 + 로봇 신호
                                    ↓ (ORB 실패 시)
                              ocr_fallback
```

### ③ 프론트엔드 아키텍처 (PyQt5 데스크톱 GUI)

> **비전문가 작업자가 마우스 클릭만으로 전체 시스템을 통제하는 3개의 탭**

| 탭 이름 | 화면 구성 | 핵심 기능 |
| :--- | :--- | :--- |
| **실시간 관제** | 좌측: 웹캠 영상 + 초록/빨강 박스, 우측: FPS 그래프 + ORB 일치율 게이지 | 현재 상태를 직관적으로 모니터링. E-Stop(긴급 정지) 버튼 배치 |
| **자가 학습** | 상단: 에러 동영상 드래그 앤 드롭 영역, 중앙: 프레임 미리보기 + 타겟 선택 드래그, 하단: "Train" 버튼 | 비전문가가 마우스만으로 데이터를 넣고 모델을 갱신 |
| **검출 이력** | 날짜별 검출 결과 테이블 (시간, 화면 번호, 확신도, 판정), 필터링/검색 기능 | 과거 데이터 열람 및 오탐 분석용 |

**GUI ↔ 백엔드 통신 구조**:
- PyQt5의 **QThread**(별도 스레드)에서 `engine/` 모듈들이 돌아가고, **Signal/Slot** 방식으로 GUI에 결과를 실시간 전송합니다.
- GUI 메인 스레드는 화면 렌더링만 전담하므로 **UI가 절대 멈추지 않습니다(Non-blocking).**

### ④ 데이터 저장 구조 (SQLite + 폴더 기반 하이브리드)

> **왜 MySQL/PostgreSQL 같은 무거운 DB가 아니라 SQLite인가?**
> 공장 로컬 PC에 DB 서버를 설치하라고 하는 순간 비전문가 작업자는 포기합니다. SQLite는 **파일 1개(`canon.db`)가 곧 DB**이므로 설치가 필요 없고, 프로그램과 함께 복사만 하면 끝입니다.

**[SQLite 테이블 설계]**

| 테이블명 | 용도 | 주요 컬럼 |
| :--- | :--- | :--- |
| `detection_log` | 매 프레임 검출 이력 | `id`, `timestamp`, `screen_id`, `confidence`, `orb_score`, `fps`, `method(ORB/OCR)` |
| `model_version` | 모델 갱신 이력 | `id`, `version`, `updated_at`, `mAP`, `file_path` |
| `error_queue` | 판별 실패 사진 큐 | `id`, `image_path`, `reason`, `reviewed(Y/N)`, `label` |

**[폴더 기반 이미지 관리]**
- 이미지 파일 자체는 SQLite에 넣지 않고 **`data/` 하위 폴더**에 저장합니다.
- DB에는 이미지의 **경로(path)**만 기록하여, 파일 시스템과 DB를 분리합니다.
- **이유**: 이미지를 DB에 직접 넣으면(BLOB) DB 용량이 기하급수적으로 커져서 로컬 PC가 뻗습니다. 경로만 저장하면 DB는 항상 가볍고 빠릅니다.
